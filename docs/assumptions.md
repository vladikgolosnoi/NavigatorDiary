# Допущения

Ниже перечислены допущения, принятые из‑за отсутствия деталей в ТЗ.

1. **Роли и принадлежность**: у пользователя одна основная роль в системе. Организатор — глобальная роль без привязки к команде; руководитель и навигаторы привязаны к одной команде.
2. **Размер команды**: ограничение 6–12 участников контролируется при подтверждении команды, но не блокирует черновую регистрацию команды до подтверждения.
3. **Статусы цели**: используются состояния `selected` → `in_progress` → `pending_confirmation` → `achieved`.
4. **Самореакции**: навигатор не может голосовать за собственную цель.
5. **Подсчёт этапов достижений**: этапы «путь/тропа/маршрут/экспедиция/успех» считаются по количеству **достигнутых** целей; этап «старт» активируется при выборе ≥6 целей.
6. **Повторный выбор целей**: повторный выбор доступен через 3 месяца, даже если предыдущие цели не достигнуты.
7. **Специальность**: одновременно допускается только одна активная специальность; новая выбирается после завершения/отмены текущей.
8. **Возрастной статус**: рассчитывается по полным годам на дату запроса.
9. **Чек‑лист специальности**: автосообщение в чате появляется после отметки всех пунктов чек‑листа выбранного уровня.
10. **Уведомления**: реализуются как записи в БД без push‑уведомлений и внешних сервисов.
11. **Справочные данные**: списки сфер/компетентностей/целей загружаются из `docs/specs/03_appendix_2_goals_catalog.pdf`, а специальности и чек‑листы уровней — из `docs/specs/04_appendix_3_specialties.docx`.
12. **Аутентификация**: в БД предусмотрены поля `email` и `passwordHash`; вход по email/паролю, если иное не будет уточнено.
13. **Ссылки на материалы**: активности и материалы в сидировании ведут на локальные файлы из `apps/web/public/resources`; видео-ссылки по специальностям ведут на поисковые выдачи YouTube по названию специальности.
14. **Регистрация пользователя**: регистрация возможна только в подтвержденную (ACTIVE) команду, иначе возвращается ошибка.
15. **Подтверждения**: подтверждение команды выполняет организатор; подтверждение пользователя выполняет руководитель своей команды.
16. **Роль при регистрации**: регистрация пользователя через публичный поток создает роль NAVIGATOR; руководитель создается отдельным процессом (админ/организатор).
17. **Формат даты следующего выбора**: API возвращает `nextEligibleAt` в ISO-формате; клиент отвечает за локализацию.
18. **Голос за цель**: один пользователь может проголосовать за цель другого навигатора только один раз; голосование доступно только навигаторам своей команды.
19. **Чек-лист специальности**: после отметки всех пунктов чек-листа у специальности заполняется `completedAt`, а подтверждение переводит статус в `COMPLETED`.
20. **Этап достижений**: если у пользователя выбрано меньше 6 целей и достигнутых целей меньше 24, этап всё равно хранится как `START` для целостности данных.
21. **Чат**: реакция на сообщение — произвольная строка (обычно emoji); WebSocket использует событие `join` с `teamId` для подписки на комнату.
22. **Анонсы**: публичный список анонсов возвращает только `isActive = true`, а `publishedAt` выставляется при публикации.
23. **Обращения**: автор и организатор могут вести переписку в обращении; статус переводится в `IN_PROGRESS` при ответе организатора и в `CLOSED` при закрытии любым из участников.
24. **Аудит**: аудит-лог хранит ключевые действия без внешних интеграций; детальная структура полезной нагрузки (`data`) минимальна и может быть расширена.
25. **Клиентская навигация**: верхнее меню и подменю соответствуют карте экранов, подменю кликабельно и ведёт к ключевым секциям экрана; верхнее меню фильтруется по ролям.
26. **Восстановление состояния**: локально сохраняется последний экран и черновик текста на каждом экране через `localStorage` для демонстрации восстановления данных.
27. **Логотипы**: использованы реальные логотипы и значки из ТЗ (блоки бренда и сфер развития).
28. **Формы доступа**: для авторизации пользователя используются поля email/пароль, для регистрации — добавлены email/пароль сверх обязательных полей из ТЗ (для согласованности с серверным API).
29. **Возраст**: в форме регистрации пользователя оставлен только нижний порог 6 лет; верхний предел не ограничивается, чтобы руководители и старшие наставники могли регистрироваться без искусственного лимита.
30. **Команды в форме**: список активных команд на регистрации навигатора загружается из публичного API `/teams/public`; при ошибке отображается сообщение, список может быть пустым.
31. **Авторизация команды**: форма входа команды остаётся упрощённой (проверка по публичному списку команд без отдельного серверного логина) и ведёт к входу руководителя.
32. **Главная/профиль**: анонсы и профиль загружаются из API; при ошибке отображаются дружелюбные сообщения.
33. **Каталог целей**: сферы/компетентности/цели загружаются из API; выбор целей и дата следующего выбора сохраняются через API, локально хранится черновой выбор.
34. **Каталог специальностей**: области/специальности/уровни загружаются из API; активна одна специальность, выбор фиксируется через API.
35. **Мои цели (UI)**: цели и активности загружаются из API, комментарии и прогресс сохраняются через API; ограничение 1 раз в неделю продублировано на UI.
36. **Мои специальности (UI)**: чек-лист и подтверждения работают через API; UI блокирует подтверждение при неверной роли и незаполненном чек-листе.
37. **Мои достижения (UI)**: возрастной статус, этап и список специальностей получаются из API.
38. **Чат (UI)**: чат интегрирован с REST и WebSocket (socket.io), подписка выполняется по `teamId`.
39. **Уведомления (UI)**: отдельный пользовательский экран уведомлений скрыт в финальной клиентской навигации; сообщения организатора доводятся через анонсы и роль-дашборды.
40. **Дополнительно (UI)**: условия использования/конфиденциальность/онлайн-консультант реализованы как рабочие блоки; онлайн-консультант отправляет обращение через API.
41. **Заметки и обращения (UI)**: функциональность на API сохранена, но отдельные пользовательские экраны скрыты из финальной мобильной навигации как не входящие в минимальную таблицу экранов.
42. **API базовый URL**: клиент использует `VITE_API_URL` (по умолчанию `/api`); для локальной разработки требуется прокси Vite или включенный CORS на сервере.
43. **Админские сценарии (UI)**: панели организатора/руководителя реализованы без заглушек; в панели организатора объединены подтверждения, анонсы, обращения и управление ресурсами участников на одной странице.
44. **Голоса за цели (UI)**: поддержка целей команды реализована в разделе «Мои цели» (блок с целями участников и кнопкой «Поддержать»).
45. **Обращения организатора (UI)**: обращения используются как механизм онлайн-консультанта в экране «Дополнительно».
46. **Интеграционные тесты API**: тесты используют мок Prisma без реальной базы данных; для полного e2e потребуется тестовая БД.
47. **Docker compose**: контейнеризация предусмотрена только для API и БД; фронтенд запускается отдельно.
48. **Демо-доступы**: для локальной демонстрации созданы тестовые пользователи (организатор/руководитель/навигатор) и активная команда.
49. **Кнопки-ярлыки**: действия верхних кнопок ведут только в реальные разделы/сценарии API без декоративных переходов.
50. **Ресурсы достижений**: в пользовательском интерфейсе веточки соответствуют статусам по этапам достижений (`START=6`, `PATH=24`, `TRAIL=48`, `ROUTE=72`, `EXPEDITION=144`, `SUCCESS=216`), жёлуди начисляются организатором за командные мероприятия, поленья — по подтверждённым специальностям (`BRONZE=10`, `SILVER=20`, `GOLD=30`); внутренние типы API (`ACORN/TWIG/LOG`) сохранены без миграции исторических данных.
51. **Значки специальностей**: в обновлённом ТЗ раздел «Значки специальностей» не содержит изображений, поэтому используются текстовые инициал‑значки (буква специальности) и единый стиль карточек.
52. **Ролевая навигация**: пункты верхнего меню скрываются для неавторизованных пользователей и ролей, которым они не нужны; защищённые экраны требуют входа и при необходимости перенаправляют на авторизацию.
53. **Навигатор (верхнее меню)**: для навигатора отображается до 6 ключевых вкладок; отдельная вкладка «Хатка бобра» скрыта, а её содержимое встроено в раздел «Мои достижения».
54. **Удаление заявок**: в соответствии с ТЗ «удаление новых пользователей/команд» реализовано как перевод статуса в `REJECTED` с фиксацией в аудит‑логе.
55. **Сообщение организаторам**: при завершении чек‑листа серебро/золото создаётся уведомление со scope `ORGANIZERS`, доступное только организаторам.
56. **Каталог целей (возраст)**: цели импортированы из файла `docs/specs/03_appendix_2_goals_catalog.pdf` в разрезе возрастной группы → сферы → компетентности → цели.
57. **Нормализация статусов**: строки `Навигатор (14-16 лет)` и `Навигатор (16-18 лет)` из файла трактуются как `Новатор` для соответствующих возрастов (очевидная опечатка), `Странник (18-22 года)` приводится к формату `Странник (18-22 лет)`.
58. **Активности для целей**: в PDF отсутствуют индивидуальные ссылки на активности, поэтому для каждой цели используется единая рабочая ссылка на каталог целей из `Приложения №2`.
59. **Каталог специальностей (Приложение №3)**: перечень специальностей и чек‑листы уровней импортированы из файла `docs/specs/04_appendix_3_specialties.docx` без смысловых сокращений.
60. **Области специальностей**: в `docs/specs/04_appendix_3_specialties.docx` нет разбивки по областям, поэтому все специальности размещены в одной области `Специальности по приложению №3`.
61. **Пересидирование специальностей**: при `prisma db seed` блок специальностей обновляется в режиме `upsert` (добавление/обновление по имени) без массового удаления пользовательских данных.
62. **Значки из PDF**: файл `docs/specs/05_appendix_badges.pdf` содержит одну страницу с набором пиктограмм; значки вырезаны по подписям и сопоставлены с этапами (`старт/путь/тропа/маршрут/экспедиция/успех`) и маскотами (`бобер/феникс`).
63. **Базовый URL API на клиенте**: если `VITE_API_URL` не задан, клиент использует относительный путь `/api`; для локальной разработки нужен Vite proxy или отдельная настройка прокси/реверс-прокси.
64. **Состав клиентских экранов**: из пользовательской навигации скрыты разделы, не входящие в минимальную таблицу экранов ТЗ (уведомления/заметки/обращения), чтобы финальный UX соответствовал утвержденной структуре.
65. **Панель организатора**: в финальной версии панели объединены подтверждения, анонсы, обращения и операции с ресурсами; отдельный экран «Хатка бобра» остаётся только как скрытый маршрут-редирект для обратной совместимости.
66. **Онлайн-консультант**: обращения создаются в экране «Дополнительно» и ведутся как переписка (ответ/закрытие) через API `appeals`; у организатора обработка обращений доступна в панели организатора.
67. **Публикация репозитория**: в Git хранятся только исходники и документация; локальные зависимости, артефакты сборки и `.env` исключены через `.gitignore`.
68. **Хранение материалов ТЗ**: исходные файлы ТЗ и приложений перенесены в `docs/specs/` с унифицированными именами для удобной навигации в GitHub.
69. **Render (стабильный cold start)**: для ускорения пробуждения free-инстанса start-команда не выполняет миграции/сиды на каждый запуск; миграции и сиды выполняются в `preDeployCommand` (один раз на деплой).
70. **Старт API при «спящей» БД**: первичное подключение Prisma к БД выполняется в фоне и не блокирует запуск HTTP-сервера, чтобы Render быстрее отдавал сайт после cold start.
71. **Главная страница**: для организатора на главной остаются логотип проекта и описание движения; для руководителя и навигатора дополнительно отображаются значки сфер развития как внешние ссылки на сайт РСДМО.
72. **Создание команды руководителем**: в панели руководителя форма «Создать команду» запрашивает только название и образовательное учреждение; населённый пункт подставляется как `Не указан`, так как в новом комментарии заказчика это поле не упомянуто.
73. **Прогресс по целям**: кликабельной является только следующая ступень шкалы; после финальной отметки цель переводится в `PENDING_CONFIRMATION`, а поддержка команды становится доступна только на этом этапе.
74. **Брендинг фонда**: логотип Фонда президентских грантов размещён в нижней части интерфейса на светлом фоне, в официальной русскоязычной горизонтальной версии без изменения пропорций и цветов.
75. **Логотипы и сферы развития**: изображения проекта и сфер развития очищены от фона и текстовых дублей; поясняющие подписи вынесены в интерфейс отдельным текстом.
76. **Аналитика организатора**: в панели организатора доступна сводка по командам, целям и специальностям, фильтр по конкретной команде, а также CSV-экспорт по командам, целям и специальностям для внешней отчетности.
